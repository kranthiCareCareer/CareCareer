# CareCareer — Canonical Identifier Strategy

Version: 1.0
Status: Phase 0 Architecture Lock
Date: 2026-07-16

---

## 1. Purpose

This document defines the identifier strategy for all CareCareer domain entities. CareCareer generates its own canonical identifiers and stores vendor/legacy IDs as external references. External system IDs are NEVER used as primary keys.

---

## 2. Identifier Format

### 2.1 Primary Key Standard

All CareCareer entities use **UUID v7** (time-ordered) as the primary key.

```typescript
// UUID v7 provides:
// - Globally unique (no collision across services)
// - Time-ordered (efficient B-tree indexing, natural sort)
// - No information leakage (unlike sequential integers)
// - Standard format (no vendor dependency)

type EntityId = string; // UUID v7 format: xxxxxxxx-xxxx-7xxx-yxxx-xxxxxxxxxxxx
```

### 2.2 Why UUID v7 Over Alternatives

| Option                            | Rejected Because                                                                 |
| --------------------------------- | -------------------------------------------------------------------------------- |
| Auto-increment integer            | Predictable, leaks count, collision across services, no multi-tenant safety      |
| UUID v4 (random)                  | Poor index performance, no natural ordering                                      |
| ULID                              | Non-standard, limited library support                                            |
| Prefixed IDs (e.g., `wkr_abc123`) | Useful for display but not as DB primary key; added as `display_id` where needed |
| Snowflake                         | Requires coordination service, adds complexity                                   |

### 2.3 Display Identifiers (Optional)

For entities that need human-friendly references, a `display_id` column is added:

```typescript
// Examples:
// Tenant: "CC-T-001"
// Worker: "CC-W-2026-00142"
// Shift: "SH-2026-07-16-0001"
// Timecard: "TC-2026-W29-00042"
// Invoice: "INV-2026-07-001"

// Display IDs are:
// - Tenant-scoped (no cross-tenant collision)
// - Sequential within tenant (for human reference)
// - NOT the primary key (UUID v7 remains PK)
// - Generated by the owning service
```

## 3. External Reference Model

Every entity that has a counterpart in an external system stores that mapping explicitly:

```typescript
/**
 * Maps a CareCareer entity to its counterpart in an external system.
 * Stored in a separate table to support many-to-many relationships
 * (one CareCareer entity may exist in multiple legacy systems).
 */
interface ExternalReference {
  /** CareCareer-generated UUID v7 */
  id: string;

  /** Owning tenant */
  tenantId: string;

  /** CareCareer entity type (e.g., 'worker', 'facility', 'shift') */
  entityType: string;

  /** CareCareer entity ID (UUID v7) */
  entityId: string;

  /** External system identifier */
  system: ExternalSystem;

  /** Entity type in the external system (may differ from CareCareer naming) */
  externalEntityType: string;

  /** The ID as it exists in the external system */
  externalId: string;

  /** When this mapping was established */
  mappedAt: Date;

  /** Whether this mapping is still active */
  active: boolean;

  /** Additional metadata (system-specific fields) */
  metadata: Record<string, unknown> | null;
}

type ExternalSystem =
  | 'symplr'
  | 'bullhorn'
  | 'labor_edge'
  | 'maestra'
  | 'auth0'
  | 'paycom'
  | 'netsuite'
  | 'hubspot';
```

### 3.1 Database Schema

```sql
CREATE TABLE external_references (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    entity_type VARCHAR(50) NOT NULL,
    entity_id UUID NOT NULL,
    system VARCHAR(30) NOT NULL,
    external_entity_type VARCHAR(100) NOT NULL,
    external_id VARCHAR(255) NOT NULL,
    mapped_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    active BOOLEAN NOT NULL DEFAULT true,
    metadata JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Unique: one entity can have only one active mapping per system+type
    CONSTRAINT uq_external_ref_active
        UNIQUE (tenant_id, entity_type, entity_id, system, external_entity_type)
        WHERE active = true,

    -- Lookup: find CareCareer entity by external ID
    CONSTRAINT uq_external_lookup
        UNIQUE (tenant_id, system, external_entity_type, external_id)
        WHERE active = true
);

-- RLS
ALTER TABLE external_references ENABLE ROW LEVEL SECURITY;
ALTER TABLE external_references FORCE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON external_references
    USING (tenant_id = current_setting('app.tenant_id')::UUID);

-- Index for reverse lookup (external → CareCareer)
CREATE INDEX idx_external_ref_lookup
    ON external_references (system, external_entity_type, external_id)
    WHERE active = true;
```

## 5. Identity Resolution Rules

### 5.1 Worker Identity (Most Complex)

A single worker may exist across multiple systems:

```
CareCareer Worker (UUID v7)
  ├── symplr.person_id = "SP-12345"
  ├── symplr.caregiver_id = "CG-67890"
  ├── bullhorn.candidate_id = "BH-11111"
  ├── maestra.worker_id = "MW-22222"
  └── auth0.sub = "auth0|abc123"
```

Resolution priority:

1. CareCareer UUID v7 is the canonical identity
2. Symplr `person_id` is the current operational identity (until cutover)
3. Bullhorn `candidate_id` is the recruiting identity (until H3 cutover)
4. Auth0 `sub` is the authentication identity (mapped, never primary)

### 5.2 Deduplication Rules

- Same person in Symplr and Bullhorn = one CareCareer Worker with two ExternalReferences
- Deduplication uses: SSN hash (restricted), name + DOB + phone, or manual operator merge
- Merge operations are audited and reversible (soft-delete the old mapping)

### 5.3 Migration Seeding

When seeding pilot data:

1. Create CareCareer entity (UUID v7 generated)
2. Create ExternalReference mapping to source system
3. Store both IDs in the external_references table
4. All subsequent queries use CareCareer UUID v7
5. Legacy lookups use the `external_references` index

## 6. ID Generation Implementation

```typescript
import { uuidv7 } from 'uuidv7';

/**
 * Generate a new CareCareer canonical identifier.
 * Uses UUID v7 for time-ordering and global uniqueness.
 */
function generateId(): string {
  return uuidv7();
}

/**
 * Generate a tenant-scoped display ID.
 * These are human-readable but NOT the primary key.
 */
async function generateDisplayId(
  tenantId: string,
  prefix: string,
  sequenceName: string,
): Promise<string> {
  // Uses a PostgreSQL sequence scoped to tenant
  const seq = await getNextSequenceValue(tenantId, sequenceName);
  const year = new Date().getFullYear();
  return `${prefix}-${year}-${String(seq).padStart(5, '0')}`;
}
```

## 7. Rules

1. **UUID v7 is the ONLY primary key format** for all CareCareer entities.
2. **External IDs are NEVER used as foreign keys** between CareCareer tables.
3. **External references are stored in a dedicated mapping table**, not inline on the entity.
4. **Display IDs are optional, tenant-scoped, and NOT the primary key.**
5. **Identity resolution across systems uses the external_references table**, not application-level string matching.
6. **Deactivating an external reference (active=false) does NOT delete the CareCareer entity.**
7. **All external reference changes are audited.**
